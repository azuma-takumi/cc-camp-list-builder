# リサーチ実行スキル

Web スクレイピングで情報を収集し、Google スプレッドシートに蓄積するスキル。
受講生(または本人)と対話しながら、**提案ベースで要件を固めてから** 実行する。

## 絶対に守るルール

1. **質問は必ずナンバリング + 箇条書き** で出す
   - 選択肢・候補・提案・最終確認すべて
   - 自由記述は最後の選択肢として「その他(自由に教えてください)」を置く
2. **いきなり実行しない**。必ず Step 1〜5 の対話を経てから実行する
3. **欲しい情報はユーザーが先に指定** → AI は「それが取れるか」のチェックに徹する
   - AI から取得項目を提案しない(候補リストはあくまで参考)
4. **最終確認(Step 5)ではスプレッドシートの列構成を必ずテーブル形式で見せる**
   - 「取得項目: A / B / C」と並べるのではなく、**実際にスプシで見えるそのままの表** で提示する
   - 各列に「想定される値のイメージ(実例)」を添える(ユーザーが完成形を脳内でイメージできる状態にする)
   - この列で集めて良いか、の yes/no を明示的に取る
5. **新しいサイトで取れない / 公式 API が必要な場合は `extend-source` スキルに委譲する**
   - ユーザーが対応したいサイト・サービスが現状のツールでカバーできない場合、自力で無理に押し通さず `.claude/skills/extend-source/SKILL.md` を読んで委譲する
   - 具体的には: 利用規約で禁止、認証が必要、公式 API の方が効率的、などのケース
6. **結果報告には必ず API 使用量(💰 ブロック)を添える**
   - `runResearch` / `enrich-contact.mjs` は結果に `usage` を含めて返す
   - Step 7 のテンプレに従い、「今回のリサーチで使った API」「今月の累計」を表形式で提示する
   - 有料 API(Google Places 等)を使った場合は金額と円換算、無料枠 API(Brave Search 等)は残量を表示する
7. **礼儀正しく**: robots.txt チェック、間隔2秒+ジッター、件数上限、Cloudflare検知で即停止は仕組みで自動化されている

## ワークフロー

### Step 1. 入口のヒアリング

ユーザーが「リサーチして」等と言ったら、まず以下を聞く:

```text
どんなリサーチをしますか? 以下のどれに近いですか?

1. 営業先リスト作成
2. 案件探索
3. 競合調査
4. その他(自由に教えてください)

具体的な内容もあれば一緒に教えてください(例: 「新宿の居酒屋の営業先リスト」)。
```

- ユーザーの回答から、**目的** と **対象の具体性** をつかむ
- 具体的な内容(地域・業種・キーワード)がまだ曖昧なら、軽く深掘りして1〜2行で言語化する

### Step 2. 対象サイト / 進め方

目的が見えたら、データの集め方を選んでもらう:

```text
進め方として、以下のどれが近いですか?

1. Google Maps から取る(飲食・美容・医療・小売など "店舗型ビジネス" の電話番号・住所・営業時間を確実に取りたい場合。Places API を使います)
2. 特定のサイトから取る(食べログ / クラウドワークス / Wantedly など、使いたいサイトが決まっている)
3. Google検索経由で幅広く(「○○ ホームページ」等で検索→ヒットしたサイトを自動巡回)
4. URLを直接渡す(すでに見つけているページがある)
5. その他(自由に教えてください)
```

**選択肢の提案ヒント**(Step 1 の目的から推測して、初期のおすすめを示せる):

- 目的が「飲食店・美容室・医療・小売などの営業先リスト(電話番号含む)」→ **1 (Google Maps)** を強く推奨
- 目的が「BtoB 企業リスト」→ 2 (Wantedly 等)を推奨
- 目的が「案件探索」→ 2 (クラウドワークス / Lancers)を推奨
- 目的が「競合調査」→ 4 (URL直接)を推奨
- 目的が「幅広い情報収集」→ 3 (Google検索)を推奨

- **1 (Google Maps)** → `mode: "places"` で実行。Step 4 はスキップして、Step 5 へ直行(取得項目は API の固定セット)
- **2** → ユーザーからサイト名か URL を聞く。URL がなければ「そのサイトのどのページから始めますか?」と聞く
- **3** → 検索クエリを聞く。AI が組み立てる場合はユーザーに提案して番号で選んでもらう
- **4** → URL を複数受け取る

> **1 (Google Maps) を選んだ場合の注意**:
> - `GOOGLE_PLACES_API_KEY` が `.env` に設定されているか事前チェック。未設定ならセットアップスキルの 2-C に案内する
> - 取得できる項目は API 固定(店名・住所・電話番号・公式サイト・評価・レビュー数・営業時間・Google Maps URL・営業状態)なので、「欲しい項目は?」の確認は省略してよい
> - 代わりに「検索クエリ」と「件数上限」だけ確認する(Step 3 / Step 5 に直接進む)

### Step 3. 欲しい情報のヒアリング

対象が決まったら、**先にユーザーから** 取りたい項目を聞く:

```text
<対象サイト>ですね。どんな情報を取りたいですか?

参考までに、<対象>でよく取られる項目はこんな感じです(あくまで候補で、選ぶ義務はありません):

1. <候補1>
2. <候補2>
3. <候補3>
...
N. その他(自由に追加してください)

番号で選ぶ、追加する、全く別の項目を指定する、どれでもOKです。
```

候補リストは、対象サイトの種類で組み立てる。例えば:

- **食べログ・ホットペッパー系** → 店名、住所、電話番号、ジャンル、価格帯、評価点数、席数、営業時間
- **クラウドワークス・Lancers** → 案件名、報酬、納期、クライアント名、カテゴリ、応募数、発注傾向
- **コーポレートサイト** → 会社名、代表者、所在地、電話番号、メール、事業内容、設立年
- **EC サイト** → 商品名、価格、在庫、レビュー数、評価、カテゴリ
- **競合調査** → 製品名、価格、特徴、ターゲット、料金プラン
- **ニュース・ブログ** → タイトル、公開日、著者、カテゴリ、本文要約、タグ

**重要**: 候補は3〜10個程度。多すぎると選びにくい。サイトの性質にフィットするものだけ。

### Step 3-P. Google Maps(places モード)の場合のヒアリング

Step 2 で「1. Google Maps から取る」を選んだ場合、こちらのフローに差し替える。
取得項目は固定なので、代わりに **検索クエリ** と **絞り込み条件** を詰める。

```text
Google Maps から取ります。以下を決めましょう:

1. 検索クエリを確認してください(Google Maps の検索窓にそのまま入れる形)
   (例:「新宿区 居酒屋」「東京都渋谷区 歯科医院」「大阪市 美容室」)

   あなたの希望を踏まえた候補:
   a. <ベストゲス1>
   b. <ベストゲス2>
   c. 自分で指定する(このチャットにそのまま入力してください)

2. 取得する項目は以下が自動でセットされます(固定):
   - 店名 / 住所 / 電話番号 / 公式サイト / 評価 / レビュー数 / 営業時間 / Google Maps URL / 営業状態

3. 絞り込みは以下が可能です。使いたいものはありますか?
   a. 今営業中のみ
   b. 評価 N 以上(例: 3.5 以上)
   c. 特に絞らない
```

> **検索クエリのコツ**(AI 側の知識として):
> - 「エリア + 業種」が最も効きやすい(例: `新宿区 居酒屋`)
> - 通称より正式な業種名が望ましい(「ヘアサロン」→「美容室」の方が結果が多い)
> - 狭すぎるクエリ(`○○駅 居酒屋`)は件数が少なくなる。駅周辺を狙うなら「エリア名+業種」+ `locationBias` で調整する

### Step 4. 取得可能性チェック(必須 / places モードはスキップ)

> **places モードの場合はこのステップをスキップし、Step 5 に直接進む**。
> (Places API が返すフィールドは固定で、取得可能性は保証されている)

ユーザーが欲しい項目を指定したら、**実際に1ページだけ取得して** 項目ごとに取れるか確認する。

まず `inspect-page` でページ構造を把握する:

```bash
node tools/research-helpers.mjs inspect-page --url "<対象URLの代表的な1ページ>"
```

出力から以下を確認:

- robots.txt で許可されているか(`robots.allowed`)
- ページ構造(見出し、JSON-LD、繰り返しクラス名など)
- 動的ページ判定(`mode` が `browser` になったか)

その上で、ユーザーが欲しい各項目について CSS セレクタの当たりをつける。
`inspect-page` には `--selectors` を渡して存在確認できる:

```bash
node tools/research-helpers.mjs inspect-page --url "..." --selectors ".address,.tel,.category"
```

結果をユーザーに報告する:

```text
<対象URL> を見てきました。結果はこうです:

1. 住所 ✅ 取得可能
2. 電話番号 ✅ 取得可能
3. ジャンル ✅ 取得可能
4. 席数 ✅ 取得可能(「店舗情報」タブにあり)
5. 客単価 ⚠️ 「¥3,000〜¥4,000」のような範囲表記で取れます(正確な単一値が必要なら後処理が要ります)

この内容で進めてよろしいですか?次のステップ(最終確認)に進みます。
```

取れないものがあれば正直に伝える:

```text
1. 住所 ✅
2. メールアドレス ❌ このページには記載がありませんでした。
   代替案として「問い合わせフォームのURL」は取れます。どうしますか?
   a. 「問い合わせフォームのURL」で代用する
   b. メールアドレスは諦めて、他の項目だけで進める
   c. その他(自由に教えてください)
```

### Step 5. 最終確認(列構成を必ず見せる)

取得項目が確定したら、実行前に **スプレッドシートに作られる列構成をそのまま** ユーザーに提示し、「この列で進めていいですか?」という形で確認を取る。

**重要**: 単に「取得項目: A / B / C」と並べるのではなく、**実際にスプシで見えるそのままの表** で見せる。ユーザーは完成後の見た目をイメージできないと判断に困る。

#### 表示フォーマット(すべてのモードで共通)

```text
📋 スプレッドシート「20260420_<名前>」に、以下の列でデータを集めます:

| 列 | 見出し | 内容 | 備考 |
|---|---|---|---|
| A | No. | 連番 | 自動 |
| B | タイトル | <例: 店名 / 会社名 / 案件名> | |
| C | URL | 詳細ページのURL | |
| D | 取得日時 | yyyy/MM/dd HH:mm:ss | 自動 |
| E | <ユーザー指定の項目1> | <想定される値のイメージ> | |
| F | <ユーザー指定の項目2> | <想定される値のイメージ> | |
| G | <ユーザー指定の項目3> | <想定される値のイメージ> | |
| ... | | | |

⚙️ 実行条件:
- 対象: <対象サイトの説明>
- モード: <list-detail / single / search-based / places>
- 件数上限: 50件
- 礼儀設定: 間隔2〜3秒、robots.txt 確認済み(許可)

この列構成とこの内容で進めてよろしいですか?

1. このまま実行する
2. 列を追加する(欲しい情報を追記してください)
3. 列を削除する(不要な列の番号を教えてください)
4. 列の順番を変える(新しい順序を教えてください)
5. 列の見出し名を変える(例: "タイトル" を "店名" に)
6. 件数を変更する
7. シート名を変更する
8. キャンセル
```

**「想定される値のイメージ」の例**(ユーザーが完成形をイメージしやすいように、具体例を1つ入れる):

| 見出し | 内容 | 備考(実例入り) |
|---|---|---|
| 住所 | テキスト | 「東京都新宿区歌舞伎町1-2-3」 |
| 電話番号 | テキスト(ハイフン含む) | 「03-1234-5678」 |
| 評価 | 数値(5段階) | 「4.2」 |
| 営業時間 | 1週間分のテキスト(スラッシュ区切り) | 「月: 17:00-24:00 / 火: 定休日 / ...」 |

取得可能性チェック(Step 4)で取れることが確認できた項目だけを列に並べる。取れなかった項目は「代替案」として Step 4 で提示したものを反映させる。

#### ユーザーが 2〜7 を選んだ場合の挙動

- **2(列を追加)** → 追加したい情報を聞く → 可能なら Step 4 に戻って取得可能性チェックをやり直し → 再度この最終確認を出す
- **3(列を削除)** → 番号で指定を受ける → 該当列を外して再表示
- **4(列の順番を変える)** → 新しい順序を受け取って反映(ただしA〜D の共通列は固定)
- **5(見出し名を変える)** → 「E列を『店名』に」のような指定を受けて反映
- **6(件数変更)** → 数字を受け取って再表示
- **7(シート名変更)** → 新しい名前を受け取って再表示(日付プレフィックスは保持することを勧める)

調整後は **必ず再度この最終確認を出す**。「変更したからすぐ実行」ではなく、一度見せ直す。

#### places モードの場合の最終確認

`mode: "places"` は列が API 固定なので、同じく列テーブルを見せて確認を取る:

```text
📋 スプレッドシート「20260420_新宿_居酒屋」に、以下の列でデータを集めます:

| 列 | 見出し | 内容 | 備考(実例) |
|---|---|---|---|
| A | No. | 連番 | 自動 |
| B | タイトル | 店名 | 「魚民 新宿東口駅前店」 |
| C | URL | Google Maps のリンク | |
| D | 取得日時 | yyyy/MM/dd HH:mm:ss | 自動 |
| E | 住所 | 詳細住所 | 「東京都新宿区歌舞伎町1-2-3」 |
| F | 電話番号 | | 「03-1234-5678」 |
| G | 公式サイト | URL | |
| H | 評価 | 数値 | 「4.2」 |
| I | レビュー数 | 数値 | 「1523」 |
| J | 営業時間 | 1週間分 | 「月: 17:00-24:00 / ...」 |
| K | GoogleMaps | リンク | |
| L | 営業状態 | 文字列 | 「営業中」「閉業」など |

⚙️ 実行条件:
- モード: places(Google Places API)
- 検索クエリ: 「新宿区 居酒屋」
- 絞り込み: <絞り込み条件 or なし>
- 件数上限: 50件

この内容で進めてよろしいですか?

1. このまま実行する
2. 検索クエリを変える
3. 絞り込み条件を変える(今営業中のみ / 評価 N 以上)
4. 件数を変える
5. 不要な列を削除する(fieldMapping で対応)
6. シート名を変える
7. キャンセル
```

places モードでは列そのものは API が返す範囲に固定されるが、「出力しない列を選ぶ」(= `fieldMapping` での除外)は可能。

### Step 6. 実行

1 が選ばれたら、config JSON を組み立てて `tools/research.mjs` で実行する。

config の組み立て方は `scrapers/_template.mjs` にパターン例がある。

#### 実行の基本形(list-detail の例)

```bash
# configを組み立てて一時ファイルに書き出し、実行
node --input-type=module <<'EOF'
import { runResearch } from './tools/research.mjs';

const config = {
  name: "新宿_居酒屋",
  mode: "list-detail",
  maxItems: 50,
  throttle: { delayMs: 2500, jitterMs: 1000 },
  list: {
    urls: ["https://example.com/list"],
    pagination: { urlTemplate: "https://example.com/list?page={page}", startPage: 2, maxPages: 3 },
    itemSelector: ".item-card",
    parseItem: {
      title: { selector: ".item-name", extract: "text" },
      url:   { selector: "a.item-link", extract: "attr", attr: "href" }
    }
  },
  detail: {
    fields: {
      住所:     { selector: ".address", extract: "text" },
      電話番号: { selector: ".tel a", extract: "text" },
      ジャンル: { selector: ".category", extract: "text" }
    }
  }
};

const result = await runResearch(config);
console.log(JSON.stringify(result, null, 2));
EOF
```

または、stdin に config JSON を流す形でもOK:

```bash
cat <<'JSON' | node tools/research.mjs --config -
{
  "name": "...",
  "mode": "list-detail",
  ...
}
JSON
```

#### 実行の基本形(places モードの例)

Google Maps から店舗情報(電話番号・住所・営業時間)を一括取得する場合:

```bash
cat <<'JSON' | node tools/research.mjs --config -
{
  "name": "新宿_居酒屋_places",
  "mode": "places",
  "query": "新宿区 居酒屋",
  "maxItems": 50,
  "placesOptions": {
    "languageCode": "ja",
    "regionCode": "jp"
  }
}
JSON
```

`placesOptions` には以下が指定可能:
- `openNow: true` — 今営業中のみ
- `minRating: 3.5` — 評価 3.5 以上
- `locationBias: { latitude, longitude, radius }` — 位置ベースの絞り込み(半径メートル)

取得される列は以下の9列(デフォルト。`fieldMapping` で上書き可能):
店名 / 住所 / 電話番号 / 公式サイト / 評価 / レビュー数 / 営業時間 / GoogleMaps / 営業状態

#### 実行中の進捗報告

長時間かかる可能性があるので、実行開始時にユーザーに伝える:

```text
開始します。件数が多い場合は数分かかることがあります。
完了したら結果を報告しますね。
```

実行が完了したら、以下のフォーマットで報告:

```text

- 重複スキップ: <N>件
- エラー: <N>件(あれば)

|---|---|---|---|
```

- `runResearch` の戻り値 `result.usage` を `formatUsageReportMarkdown()` に渡すとこの表が返る
- AI はそれをそのまま(または多少の文言補正をして)ユーザーに表示する
- 有料 API が登場しなかった場合でも「有料 API の使用なし」+ 「今月の累計」は出す

**4 を自発的に強く推したほうが良い条件**:
- `list-detail` / `search-based` モードで、電話番号列や連絡先系の列が **空または未取得** のとき
- BtoB 系のサイト(Baseconnect / Wantedly / OpenWork / Aperza 等)から取得したとき
- ユーザーが「営業先リスト」と言っていたのに、電話番号がほとんど入っていないとき

この場合、選択肢を出すより先に「電話番号が空の行が多そうです。`enrich-contact` で補完すると効率的です」と一言添えてから 4 を選んでもらう形にする。

## 特殊ケースの対応

### 「よく分からないから任せたい」モード

ユーザーが「適当に任せて」「おすすめで」等と言ったら、**最低限の確認** だけ取る:

```text
了解です。最小限の確認だけさせてください。

1. 対象サイトは <ベストゲス候補> でいいですか? それとも別のサイト?
2. 件数の上限は 50 件でいいですか? それとも別の数?

この2つだけ決まれば、あとは私が一般的なベースセット(名前・住所・電話・カテゴリ)で進めます。
```

### robots.txt で禁止されている場合

```text
⚠️ このサイトは robots.txt で <該当パス> のアクセスが禁止されています。
(具体的なルール: <Disallow: /xxx>)

以下のどれで進めますか?

1. 別のサイト・別のページで試す
2. リサーチを中止する
3. 自己責任で実行する(サイトに負担をかけない範囲で、件数少なめにすれば問題になりにくいが、規約違反のリスクあり)
```

### Cloudflare / Captcha 検知

スクレイピング実行中に検知されたら、仕組みが自動停止してエラーを返す。ユーザーには:

```text
⚠️ <対象サイト> のアンチボット(Cloudflare / CAPTCHA)が検知されました。
ここまでで <N>件 取得できています。

以下のどれで進めますか?

1. ここまでの結果で確定する
2. 手動取得に切り替える(ユーザーが URL をコピペで渡す形)
3. 別のサイトから情報を補完する
4. 終了
```

### HTTP 429 / レート制限

自動で `Retry-After` を尊重してリトライするが、連続でダメなら停止する。

## 連絡先エンリッチ(電話番号・メールを後追加する)

Step 7 で選択肢 4 が選ばれた場合、または「電話番号を足して」「連絡先を補完して」「エンリッチして」等と言われた場合は `tools/enrich-contact.mjs` を使う。

### 前提チェック

1. 対象シートの **URL 列(既定: `公式サイト`)に URL が入っているか** を `list-sheets` or `readSheet` で確認
2. BtoB 系で Places API でもフォールバックしたい場合は、`GOOGLE_PLACES_API_KEY` が設定されているか確認(未設定なら Places フォールバック抜きで進める)

### 対話フロー(簡潔版)

```text
電話番号・メールを補完します。以下を確認させてください:

1. 対象シート: <シート名>(リサーチ直後ならそのまま使います)
2. URL 列の見出し名: 公式サイト / <別の列名> / 自動で見つける
3. Places API でのフォールバックを使いますか?
   a. 使う(スクレイピングで見つからなかった行だけ Places で検索して補う)
   b. 使わない(スクレイピングのみ)
   c. Places API のみ(スクレイピングをスキップし、検索のみで埋める = 店舗型ビジネスに有効)
4. まず何件で試しますか?
   a. 5件(動作確認)
   b. 全件
   c. 件数を指定(数字で答えてください)
```

### 実行

ユーザーが選んだ内容を元に実行する:

```bash
# パターン A: BtoB 企業 → scrape + Places フォールバック
node tools/enrich-contact.mjs --sheet "<シート名>" --url-column "公式サイト" --use-places

# パターン B: スクレイピングのみで試す
node tools/enrich-contact.mjs --sheet "<シート名>" --url-column "公式サイト"

# パターン C: 店舗型 / 既にリストはあるが URL がない場合 → Places のみ
node tools/enrich-contact.mjs --sheet "<シート名>" --places-only

# 動作確認用(先頭 5 件だけ)
node tools/enrich-contact.mjs --sheet "<シート名>" --url-column "公式サイト" --use-places --max 5
```

### 結果報告

```text

- 対象: <N>行
- 電話 or メールが取れた行: <M>行 (ヒット率 <X>%)
- うち Places で救えた行: <P>行(`--use-places` のとき)

追加された列: 「電話番号(推定)」「メール(推定)」
(既存の値は上書きしていません)

シート: https://docs.google.com/spreadsheets/d/<SPREADSHEET_ID>/edit#gid=<sheetId>

💰 今回のエンリッチで使った API

| API | リクエスト | 推定費用 | 備考 |
|---|---|---|---|
| Google Places API (New) | 28 req | $0.896(約 134円) | 月 $200 無料クレジット内 |

ℹ️ 今月(2026-04)の累計

| API | リクエスト | 推定費用 | 備考 |
|---|---|---|---|
| Google Places API (New) | 70 req | $2.240(約 336円) | 月 $200 無料クレジット内 |

※ USD→JPY 換算は 150 円/USD(`.env` の `USD_JPY_RATE` で変更可)

次にどうしますか?
1. 見つからなかった行を目視で確認したい
2. 全件に対して Places フォールバックだけ追加で回す(--places-only)
3. 別のリサーチを始める
4. 終了
```

**💰 ブロックについて**: `node tools/enrich-contact.mjs` は標準出力の末尾に `formatUsageReportMarkdown()` の出力を自動で付けているので、AI はその内容をユーザー向けに整形して見せればよい。

### 注意事項

- **既存の値は上書きしない** — 電話・メール列に既に値が入っている行はスキップされる
- **結果は「(推定)」列に書き込まれる** — 既存の手作業値と競合しないよう、新規列として追加される
- **Places フォールバック時は `会社名+住所` でマッチング** — 同名の別支店を誤って拾わないよう、住所の市区町村レベルで照合している
- **`--places-only` は注意** — URL 列がなくても会社名だけで動くが、同名別会社が混ざるリスクが高い。候補が1件だけ返ったときのみ採用する安全ロジックは入っているが、重要案件では結果を必ず目視確認する

## よくあるミス(避けるべき挙動)

- 🚫 いきなり実行してしまう → 必ず Step 1〜5 を踏む
- 🚫 AI が取得項目を先に提案してしまう → ユーザーに先に聞く
- 🚫 質問を文章で出してしまう → 必ずナンバリング+箇条書き
- 🚫 取得可能性チェックを飛ばして実行する → 少なくとも1ページは事前に見る
- 🚫 「200件取ってきた」と言いつつ実は robots.txt 禁止の場所だった → 最初に必ず confirmRobots
- 🚫 エラーを隠す → 失敗した件数・理由は正直に伝える
