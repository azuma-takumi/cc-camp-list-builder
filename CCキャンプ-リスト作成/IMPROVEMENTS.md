## 案件概要

- 案件名: `CCキャンプ-リスト作成`

## 詳細メモ

### 1. URL品質の改善

- 背景: 日本語を含む YouTube URL や handle URL で、開けない・再現しにくいものがあった
- 対応:
  - URLチェックと軽い自動修正を標準化
  - YouTube URL は `channel/UC...` 形式に正規化
  - 専用スクリプトで対象列を固定し、誤列更新を防止
- 学び:
  - URLは保存時に正規化した方が後工程が安定する
  - 列番号はマジックナンバーにせず定数化した方が安全

### 2. 会社名・代表者名補完の改善

- 背景: 代表者名に「代表取締役」などの役職が混ざり、会社名も拾い過ぎることがあった
- 対応:
  - 役職除去ロジックを導入
  - 信頼性チェックで短すぎる値やノイズを除外
  - 既存行を再評価する補完スクリプトを用意
- 学び:
  - 「抽出」と「整形」を分けると改善しやすい
  - 補完系は既存行を再走査できる形にしておくと運用が楽

### 3. メアド取得の改善

- 背景: 通常の問い合わせページだけではメール欄が埋まらないことがあった
- 対応:
  - 特商法ページ、会社概要、問い合わせ導線を優先探索
  - メール取得元をシートで選べるように整理
- 学び:
  - 「どこから取ったか」も一緒に残すと再確認しやすい
  - 取得元の分類は後から集計や見直しに効く

### 4. APIクォータ管理

- 背景: YouTube API の上限到達で処理が止まることがあった
- 対応:
  - 複数キーのフォールバック
  - 高コストな `search.list` 依存を減らし、低コスト API を優先
  - 使用量サマリーを出力
- 学び:
  - 精度だけでなく、API単価も設計対象
  - 「なぜ止まったか」を可視化しないと次回改善につながらない

### 5. シート更新の安全性

- 背景: A列の数式や色付きセルの仕組みを壊すと、手戻りが大きい
- 対応:
  - `A列は触らない`
  - `東たくみ` 以外の行は変更しない
  - 更新対象列を明示したスクリプトに分離
- 学び:
  - シート案件は「何を更新するか」より「何を触らないか」を先に決めるべき
  - dry-run か限定範囲指定があると事故が減る

### 6. ログ要約の標準化

- 背景: ログファイルは残っていても、スクリプトごとにサマリーの書き方が違い、見返しづらかった
- 対応:
  - 共通のサマリー書き出しヘルパーを追加
  - `生成日時` `ステータス` `概要` `指標` を共通見出しとして統一
  - URLチェック系、テスト系、YouTube探索系のサマリーを同じ型に寄せた
- 学び:
  - 長いログとは別に「最初に見る1枚」を固定すると作業再開しやすい
  - 共通フォーマットは実装より先に見出しを決めた方が広げやすい

### 7. 日本語 YouTube URL の混入（2026-04-19）

- 背景: `searchYoutubeChannels` がチャンネルの `customUrl`（ハンドル）をそのまま URL に埋め込む実装になっていた。YouTube API は日本語ハンドルもそのまま返すため、`@みおんゴルフチャンネル` のような非ASCII URLがシートに書き込まれた
- 対応:
  - `youtube-api.mjs` で非ASCIIハンドルは `channel/UC...` 形式にフォールバック（root fix）
  - 既存の6行を `ONLY_NON_ASCII=1 node tools/normalize-youtube-channel-urls.mjs` で修正
- 学び:
  - YouTube API の `customUrl` は日本語になりうる。URL 書き込み前に必ず ASCII チェックが必要
  - 修正専用スクリプト（`normalize-youtube-channel-urls.mjs`）が既にあったので事後対応は容易だった

### 8. 「ショップ名」ラベルで会社名が取れなかった（2026-04-19）

- 背景: ティーオリーヴ・ラケットハウスキング等、特商法ページや概要欄に「ショップ名：xxx」と書いてある店舗で会社名が空欄になった。`COMPANY_PATTERNS` が「ショップ名」「店名」「屋号」を認識しない設計だった
- 対応:
  - `contact-discovery.mjs` と `backfill-company-info.mjs` の `COMPANY_PATTERNS` に「ショップ名・店舗名・店名・屋号」を追加
- 学び:
  - 法人格（株式会社）を持たない個人店やショップも営業リスト対象になりうる。「会社名」以外の表記もパターンに入れておく
  - 未取得行は手動確認で発見できるが、都度パターンを拡張していく姿勢が大事

### 9. 会社名フィールドの後置ノイズ（2026-04-19）

- 背景: スクレイプで取れた会社名に「社名：株式会社xxx」「株式会社xxx（英語表記）」「株式会社xxx　お問い合わせページ」等のノイズが入った。`sanitizeCompanyName` だけでは取り切れなかった
- 対応:
  - `backfill-company-info.mjs` に `cleanDirtyCompanyName()` を実装
  - プレフィックス（「社名：」「商号：」等）を除去、括弧内の英語・カタカナ別表記を除去、後置ノイズ（「お問い合わせページ」等）を除去
  - `(株)` は `株式会社` に変換
- 学び:
  - スクレイプは「取れた値」を素直に使うとノイズが混入する。取得直後にクリーニング関数を通す習慣をつける
  - dry-run で before/after を確認してから書き込む手順が重要（今回この手順で誤修正を防げた）

### 10. 日付列（J・K）の左揃え混入（2026-04-20）

- 背景: オリジナルシートのJ・K列の中央揃え書式は**セル単位**で設定されており、列全体のデフォルトではない。`updateRows()`（`spreadsheets.values.update`）は値のみ書き込み、書式は一切設定しない。そのため、書式のなかった新規セルに値を書き込むと左揃えになった。行156・168・169・173・177の5件で発生
- 対応:
  - `batchUpdate` で該当5行のJ・K列に `horizontalAlignment: CENTER` / `verticalAlignment: MIDDLE` を明示的に設定
  - オリジナルへの転記スクリプトに、値書き込み後に書式を適用するステップを追加
- 学び:
  - `values.update` は書式を保持も設定もしない。既存セルへの上書きは既存書式を引き継ぐが、空セルへの書き込みは書式なし（左揃え）になる
  - テンプレートと同じ見た目にするには、値の書き込みとは別に `batchUpdate` で書式も明示的に設定する必要がある
  - 転記後は目視でなくスクリプト（`effectiveFormat.horizontalAlignment` チェック）で書式を確認する習慣をつける

### 12. チャンネル登録者数・最終投稿日の書き込み漏れ（2026-04-20）

- 背景: discoverスクリプト・backfillスクリプトともに登録者数（I列）・最終投稿日（J列）を空文字で書き込んでいた。後から `fill-youtube-channel-metrics.mjs` で補完する運用になっていたが、転記後に補完を忘れるリスクがある
- 対応: `discover-investment-leads.mjs` / `backfill-investment-emails.mjs` / `discover-youtube-sales-leads.mjs` の3スクリプトで、メール取得後に `getYoutubeChannelMetricsByUrl` を呼び出し、書き込み時点で I・J列を埋めるよう修正
- 再発防止: RULES.mdに「行を書き込む時点でメトリクスを必ず取得する」ルールを追加。別途補完スクリプトを走らせる運用は廃止

### 11. オリジナルシートの書き込み位置ずれ（2026-04-20）

- 背景: `findStartRow` は「B列に値がある最終行の次」を返す実装だが、別記入者が途中の行をスキップしてずっと後ろ（row1000付近）から書いていた。そのため row322 以降が空いているにも関わらず row1095 に書き込んでしまった
- 対応: row1095〜1119をクリアし、row322から書き直し
- 再発防止: 転記前に書き込み開始行の周辺（前後5行）を必ず確認し、想定外に大きい行番号の場合はシート全体をスキャンして最初の空き行を特定する

### 13. 不正チャンネルが収集後に再追加される問題（2026-04-21）

- 背景: 収集スクリプトはシート上の既存チャンネル名をもとに重複除外するため、一度削除した行は「存在しないチャンネル」として再追加される。同じクエリを再実行すると同一の不正チャンネル（住宅無関係・サンプルメール等）が再書き込みされた
- 対応:
  - `discover-housing-consulting-leads.mjs` に `CHANNEL_BLACKLIST`（Set）と `SAMPLE_EMAIL_PATTERN`（regex）を追加
  - 候補取得時・メール確定時の2か所でフィルタリング
- 再発防止: 不正と判明したチャンネルは削除するだけでなく、スクリプトの `CHANNEL_BLACKLIST` にも必ず追加する

### 14. 住宅コンサルティング業界リスト初回収集完了（2026-04-21）

- 実施内容: `discover-housing-consulting-leads.mjs` を新規作成し、60以上のクエリで住宅・不動産・リノベ系YouTubeチャンネルを収集

  - JICPAメール（`kinki@jicpa.or.jp`）- チャンネルオーナー本人のアドレスでない

### 15. チャンネル名末尾空白による重複除外バイパス（2026-04-21）

- 背景: YouTube APIが返すチャンネル名に末尾空白が混入することがある（例: `"柊の住処   "`、`"現役バリバリ社長の家づくりch  "`）。`existingChannelNames` はtrim済みで構築されるが、`existingChannelNames.has(candidate.channelName)` の比較時にcandidateをtrimしていなかったため、同一チャンネルが繰り返し追加された
- 対応: `discover-housing-consulting-leads.mjs` と `discover-investment-leads.mjs` の重複チェック箇所を `candidate.channelName.trim()` に修正
- 修正実績: 2回の収集で `柊の住処`・`現役バリバリ社長の家づくりch` が再追加 → 都度削除で対処。根本修正後は再発なし
- 再発防止: candidateのチャンネル名は比較前に必ずtrimする。existingNamesの構築時だけでなく参照時も同様

### 16. YouTubeページ構造変更によるスクレイプ破損（2026-04-22）

- 背景: YouTube /aboutページのJSONデータ構造が変更され、`channelAboutFullMetadataRenderer`・`channelHeaderLinksRenderer` が消滅。`deepScrapeYouTube` の説明文取得・サイトURL取得が両方ともnullを返すようになった
- 対応:
  - 説明文: `channelMetadataRenderer.description` を最優先し、旧キーをフォールバックとして残す
  - サイトURL: `channelExternalLinkViewModel` が使うYouTubeリダイレクトURL（`youtube.com/redirect?q=...`）を正規表現で全データJSON文字列からスキャンして抽出
  - `backfill-sports-emails.mjs`・`backfill-investment-emails.mjs` 両方に同一修正を適用
- 再発防止: YouTube APIを使わないスクレイプはページ構造変更でサイレントにnullを返す。定期的に既存チャンネルでテストして構造変更を早期検知する

### 17. API全枯渇時のWebSearchフォールバック収集（2026-04-22）

- 背景: YouTube API（3キー）・Brave API・Google Custom Search API が全て同日に枯渇。通常の自動バックフィルスクリプトが0件しか収集できなくなった
- 対応:
  - Claude組み込みのWebSearchツールで住宅・工務店系YouTubeチャンネルを直接検索
  - 記事ページ（gotta-ride.com等）のチャンネルリストをスクレイプして候補URLを収集
  - 各チャンネルの/aboutページ→公式サイト→contactページの順でメールを探索
  - 有効リード13件を発見（Regard、縁側家族、松尾設計室、住宅不動産チャンネル等）
- 再発防止: API枯渇が重なった場合のフォールバック順序を明確化: ① Brave → ② Google CSE → ③ WebSearch+手動スクレイプ → ④ 探索ログの残余チャンネルをバッチスクレイプ
