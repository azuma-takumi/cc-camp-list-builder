# プロジェクト構築スキル

ユーザーの「こんなの作りたい」を、対話しながら動くプロジェクトに仕上げる。

---

## 全体フロー

```
開始 → ヒアリング → UX設計 → 要件定義 → 環境準備 → 実装 → 動作チェック → 調整 → 完成
```

各ステップで必ずユーザーの確認を取ってから次に進む。

---

## Step 1: ヒアリング

ユーザーに以下を聞く。一度に全部聞かず、会話のテンポに合わせて1〜2問ずつ。

### 必ず聞くこと

1. **何を作りたいか**（ざっくりでOK）
2. **誰が使うか**（自分だけ？他の人にも？）
3. **どんなデータや機能が必要か**

### 必要に応じて聞くこと

4. どのくらいの頻度で使うか
5. 今はどうやっているか（既存のワークフロー）
6. 完成イメージに近いもの（参考サービス・ツール等）

### ヒアリングの深さ

ユーザーの理解度に合わせる:
- 明確に説明できる人 → さっと進む
- ふわっとしている人 → 具体例を出して一緒に考える

### ヒアリング時の案内例

```text
まず、どんなものを作りたいか教えてください。
ざっくりで大丈夫です。たとえば:

- スプレッドシートにデータを入れたら自動で処理してほしい
- 簡単なWebアプリを作りたい
- APIから情報を取得して管理したい
- AIを使った自動化ツールを作りたい

どんなことがしたいですか？
```

---

## Step 2: ユーザー体験設計

ヒアリング結果をもとに、ユーザーがどう操作するかを一緒に決める。

### やること

1. **操作フローを言葉で整理する**
   - ユーザーが何をして → システムが何をして → 結果がどうなるか
2. **データの流れを決める**
   - 入力は何か、出力は何か、どこに保存するか
3. **ユーザーが操作する方法を決める**
   - チャット（Cursor AIチャット）で操作するか
   - Webブラウザで操作するか
   - スプレッドシートで操作するか
   - コマンドで実行するか

### 提示例

```text
整理すると、こんな流れになりそうです:

【操作フロー】
1. あなたが ○○ する
2. ツールが △△ を実行する
3. 結果が □□ に出力される

【データの流れ】
入力: ○○
処理: △△
出力: □□

この方向で進めてよいですか？
```

---

## Step 3: 要件定義

スコープと技術選定を明確にする。

### やること

1. **やること**リストを作る
2. **やらないこと**リストを作る（今回のスコープ外）
3. **技術選定**を提案する
4. ユーザーに確認して合意を取る

### 技術選定の原則

- **最もシンプルな方法を選ぶ**。同じことを実現できる方法が複数あるなら、より簡単な方
- **ユーザーの経験レベルに合わせる**。初心者なら枯れた技術、経験者なら最適な技術
- **段階的に作る**。最初はMVP（最小限で動くもの）、あとから拡張

### 提示例

```text
では、今回作るものの範囲を決めましょう。

【やること】
- ○○ の機能
- △△ の機能

【やらないこと（今回のスコープ外）】
- □□
- ◇◇

【技術選定】
- ○○: △△を使います（理由: シンプルで始めやすい）
- □□: ◇◇を使います

これでよければ、環境準備に進みます。
変えたいところがあれば教えてください。
```

---

## Step 4: 環境準備

プロジェクトに必要な環境をセットアップする。

### 4-1: 何が必要か判断する

要件定義の結果に基づいて、必要なセットアップを洗い出す:

- **Google API を使う場合** → `setup/SETUP-FOR-CC.md` の手順でGCP認証
- **外部APIを使う場合** → APIキーの取得を案内
- **デプロイする場合** → デプロイ先のアカウント作成を案内
- **特に不要な場合** → スキップ

### 4-2: プロジェクト名を決める

```text
プロジェクトの名前を決めましょう。
英語の小文字とハイフンで、短くわかりやすい名前がいいです。

例: reading-log, task-tracker, my-portfolio

何にしますか？
```

### 4-3: プロジェクトディレクトリを作成

cc-project-builder/ と同じ階層に新しいディレクトリを作成する。

```bash
mkdir -p ../{project-name}
```

---

## Step 5: 実装

プロジェクトファイルを生成する。

### 実装の原則

1. **まずMVP**を作る。全機能を一度に作らず、最小限で動くものを先に完成させる
2. **1ファイルずつ作る**。作ったらユーザーに何を作ったか簡潔に説明する
3. **コードはシンプルに**。1関数あたり20行以内を目安。小さく分ける
4. **日本語コメントを入れる**。各関数の冒頭に「何をする関数か」を日本語で書く
5. **エラーハンドリングは丁寧に**。何が起きたか・どうすればいいかを日本語で出力

### スプシ連携プロジェクトの場合

`references/` フォルダにある実装例を参考にする。
ただし、そのままコピーではなく、ユーザーの要件に合わせてカスタマイズする。

参考にする部分:
- `auth-google.mjs.example` → Google OAuth 認証フロー
- `sheets.mjs.example` → Sheets API ヘルパー関数
- `create-template-sheet.mjs.example` → テンプレートスプシ作成
- `CLAUDE.md.example` → プロジェクト定義ファイルの構成
- `SKILL.md.example` → スキル定義の構成
- `SETUP-FOR-CC.md.example` → セットアップ手順書の構成

### それ以外のプロジェクトの場合

要件に合わせてゼロから構築する。
ただし、以下の構成は全てのプロジェクトで共通:

- **CLAUDE.md** — プロジェクトの全体定義（何をするツールか、コマンド一覧、構成図、スキル定義）
- **.cursor/skills/{skill-name}/SKILL.md** — メインのスキル定義
- **package.json** or 適切な依存管理ファイル
- **.gitignore** — 機密情報を除外
- **.env** + **.env.example** — APIキー等の環境変数（必要な場合）

### 実装中の進捗報告

大きなプロジェクトでは、区切りごとにユーザーに進捗を報告する:

```text
進捗報告:
✓ プロジェクトの骨格を作成しました
✓ メイン処理のスクリプトを作成しました
→ 次は動作テストです
```

---

## Step 6: 動作チェック

生成したプロジェクトが実際に動くか確認する。

### やること

1. 必要な依存関係をインストール
2. テスト用のデータを用意（ユーザーに入力してもらう or ダミーデータ）
3. メイン処理を実行
4. 結果を確認して報告

### エラーがあった場合

1. 原因を特定する
2. ユーザーに「○○が原因でした」と平易に説明する
3. 修正して再実行する

---

## Step 7: 調整

ユーザーの「ここをこうしたい」に対応する。

### 対応の原則

- 1つの変更ずつ対応する
- 変更後は必ず動作確認する
- 複雑になりそうな場合は「シンプルにするとこうなります」と代替案を提示する
- ユーザーが「いや、こうしたい」と言えばその方向で進める

---

## Step 8: 完成

動作確認が終わったら、完成を宣言する。

### 8-1: まとめを提示

```text
プロジェクトが完成しました！

## {project-name} — {ツールの説明}

### 使い方
（操作手順を簡潔に）

### ファイル構成
（生成したファイルの一覧と各ファイルの役割）

### 困ったときは
- エラーが出たら、そのままこのチャットに貼り付けてください
- CLAUDE.md にツールの全情報が書いてあるので、AIが理解して対応できます
```

### 8-2: プロジェクトの CLAUDE.md にエラー対処法を記載

動作チェック中に発生したエラーがあれば、生成したプロジェクトの CLAUDE.md に「よくあるエラーと対処法」として追記する。

### 8-3: 次にできることの案内

```text
今後、このツールを改良したいときは:
- プロジェクトフォルダを Cursor で開いて、やりたいことをAIチャットに伝えてください
- CLAUDE.md にツールの全情報が書いてあるので、AIが理解して対応できます
```
