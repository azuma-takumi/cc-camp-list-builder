# 改善ログ

この案件で実施した改善と、他案件にも横展開できる学びを残すメモです。

## 案件概要

- 案件名: `自作-リスト作成(水、化粧品、サプリ)`
- 主な対象: TV、自社通販、Yahoo、楽天を横断した営業リスト収集とシート整理
- 主な出力先: Google スプレッドシート

## 改善一覧

| テーマ | 問題 | 対応 | 効果 | 横展開 |
|---|---|---|---|---|

| シート間重複 | 同じブランドが複数シートに残る | ブランド名正規化 + 優先度ベースの重複削除 | 重複行を安定して削除できた | 横断統合案件 |

## 詳細メモ

### 1. 収集順序と優先度の固定

- 背景: TV、自社通販、Yahoo、楽天を並行で扱うと、どれを残すかの判断がぶれやすい
- 対応:
  - `TV > 自社 > Yahoo > 楽天` の優先順位をコード化
  - 全体実行後に重複整理を入れる流れに統一
- 学び:
  - 先に「残す基準」を決めると後処理が安定する
  - 優先順位はコメントだけでなくコード定数に置いた方がブレない

### 2. シート間重複の自動整理

- 背景: 同じブランドが複数シートに存在し、営業対象が二重管理になりやすい
- 対応:
  - ブランド名を正規化してキー化
  - 優先度の低いシート側を削除
  - 削除は後ろから行い、行ズレを防止
- 学び:
  - シート削除は逆順実行が基本
  - 重複判定は表示名そのままではなく、比較用キーが必要

### 3. Yahoo 問い合わせURLの正規化

- 背景: 問い合わせURLがバラバラで、シート上の問い合わせ先品質が不安定だった
- 対応:
  - 店舗URLから store ID を抽出
  - `https://talk.shopping.yahoo.co.jp/contact/{store_id}` に再構築
  - dry-run を用意してから反映
- 学び:
  - 入力値から正規URLを再構築できるなら、その方が安定する
  - 本更新前に dry-run を用意すると安心して実行できる

### 4. 個別修正の明文化

- 背景: 一部ブランドだけ連絡先やURLが例外的で、都度手直しになりやすい
- 対応:
  - URLごとの修正内容を `FIXES` に集約
  - 追加候補も `PENDING_ADD` として明文化
- 学び:
  - 例外はその場しのぎにせず、マップ化すると再利用できる
  - 「修正」と「追加候補」を分けると管理しやすい

### 5. ログと長時間バッチの可視化

- 背景: 長時間処理で、あとから進行状況や失敗箇所を追いにくい
- 対応:
  - stdout と同内容をファイルにも出力
  - ログファイルを時刻付きで保存
  - 必要ならログファイル出力を無効化できるようにした
- 学び:
  - 長い収集処理は「成功したか」より「どこまで進んだか」が重要
  - ログのオンオフ切り替えがあると使い分けしやすい

### 6. 削除系スクリプトの安全策

- 背景: クリーンアップで必要行やサンプル行を消すリスクがある
- 対応:
  - サンプル行は絶対に触らない
  - キーワード検証後に削除
  - 削除は逆順バッチで実施
- 学び:
  - 削除処理は「守る範囲」を先に固定するべき
  - いきなり消すより、削除条件のログ出しが重要

### 7. ログ要約の標準化

- 背景: 実行ログは残っていても、`Yahoo` `楽天` `TV` `main` で見返し方がそろっていなかった
- 対応:
  - `latest-summary.md` を出す共通ヘルパーを追加
  - `main.mjs` `scrape-yahoo.mjs` `scrape-rakuten.mjs` `scrape-tv.mjs` に要約出力を追加
  - `生成日時` `ステータス` `概要` `指標` を共通見出しに統一
- 学び:
  - 進捗ログと結果サマリーは別物として持った方が運用しやすい
  - 入口スクリプトだけでなく、単体実行スクリプトにも同じ要約があると再利用しやすい

### 8. Yahoo カテゴリ事後検証と収集時フィルタ

- 背景: ウォーターサーバー・化粧品・サプリのシートに、実際にはその商品を扱わない無関係ショップが多数混入していた
- 原因:
  - 広いクエリ（例: `プロテイン サプリ`→スポーツ店、`宅配水`→飲料店）で検索すると、商品ページがあっても専門店でないショップが引っかかる
  - 収集時にカテゴリ商品の有無を確認する仕組みがなかった
- 対応:
  - **事後クリーニング**: `verify-category-stores.mjs`（Yahoo）・`verify-water-server-stores.mjs`（Yahoo ウォーター）・`verify-rakuten-stores.mjs`（楽天）でストア内検索/API検証を実施して削除
  - **再発防止**: `scrape-yahoo.mjs` の収集ループ内で `verifyStoreHasCategory()` を呼び出し、0件なら追記スキップ
  - **楽天**: `scrape-rakuten.mjs` で `fetchShopItemCountForVerify()` を収集時に呼び出しスキップ
- 削除実績: Yahoo 計181件 / 楽天 計104件
- 学び:
  - 収集と検証を分離すると後で大量クリーニングが必要になる。収集ループ内に「この店は本当にその商品を扱うか」チェックを入れておくのが一番コスパが良い
  - Yahoo のストア内検索の廃止URLに注意: `/search/?p=` は 489 エラー、正しくは `/search.html?p=`

### 9. 楽天 API の shopCode フィルタ活用

- 背景: 楽天は API で収集するため、検証に別のアプローチが必要と思われた
- 発見: `IchibaItem/Search` API に `shopCode=xxx` を追加するだけで、そのショップの対象商品数が返る
- 対応:
  - `fetchShopItemCountForVerify({ shopCode, keyword })` を `rakuten-webservice.mjs` に追加
  - 収集時と事後検証の両方で同じ関数を流用
- 学び:
  - 既存 API に 1パラメータ追加するだけで別機能を実現できる場合がある。二度手間を恐れる前に API ドキュメントを確認する

### 10. 外部 API の 429 対策（指数バックオフ）

- 背景: 楽天 API を連続呼び出しすると HTTP 429 Too Many Requests が多発し、全検証の30%以上がエラースキップになっていた
- 対応: 呼び出し関数内に `backoffs = [0, 3000, 8000, 20000]` のリトライを追加
- 効果: 429 エラーが0件になり、全件検証が確実に完了するようになった
- 学び:
  - 外部 API を大量呼び出しするスクリプトは最初からバックオフリトライを入れておく
  - 遅延は「固定 + ランダム揺らぎ」にするとリクエストが分散しやすい

## 今後この案件で標準化したいこと

- 主要補正スクリプトに dry-run をそろえる
- 収集後の自動監査フローを固定する
- 例外修正マップの保守ルールを決める
- 個別補正スクリプトにも要約ファイルを広げる
