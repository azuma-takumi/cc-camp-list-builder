# 営業リスト案件 横断の改善・学び一覧

対象案件:

- `cursor/リスト作成/CCキャンプ-リスト作成`
- `cursor/リスト作成/自作-リスト作成(水、化粧品、サプリ)`

このファイルは、案件固有のメモではなく、複数案件で再利用できる改善と学びを一覧化するためのものです。

## 採用済み運用ルール

- URLは保存前に正規化する
- シート更新前に、対象列・対象行・非対象領域を明記する
- 更新系・削除系スクリプトには dry-run か範囲指定を付ける
- 例外修正はマップ化してコードに残す
- ログは stdout だけでなくファイルにも残す
- 実行後にまず見るサマリーの見出しを統一する
- 行削除は後ろから実行する

### 保留項目の進め方メモ

- `高`: 次の改善候補として優先的に着手
- `中`: 高優先が一段落したあとに共通化
- `担当案件`: まずどちらの案件から着手すると効果が大きいか
- `実装候補ファイル`: 入口として最初に見に行く場所

## テーマ別メモ

### URL正規化

- 代表例:
  - YouTube URL を `channel/UC...` に統一
  - Yahoo 問い合わせURLを store ID から再構築
- 学び:
  - 元の入力をそのまま保存するより、比較・再利用しやすい形式に直した方が安定する

### 重複整理

- 代表例:
  - `TV > 自社 > Yahoo > 楽天` の優先度で重複削除
  - 同一企業・同一チャンネルの判定条件を先に決める
- 学び:
  - 重複削除はアルゴリズムより先に「残す基準」の定義が必要

### 例外管理

- 代表例:
  - URLごとの個別 contact 修正
  - 補完時の例外ワードや非信頼値の除外
- 学び:
  - 例外をコードに見える形で残すと、次案件への移植がしやすい

### シート安全性

- 代表例:
  - `A列は触らない`
  - 特定の記入者の行だけ更新
  - 更新対象列を固定
- 学び:
  - 更新ロジックは「何を触るか」より「何を触らないか」が重要

### dry-run と限定実行

- 代表例:
  - `--dry-run`
  - `START_ROW` / `END_ROW`
  - `--phase1-only`
- 学び:
  - 一括更新案件では、小さく試してから本実行する導線が必須

### ログ可視化

- 代表例:
  - 実行ログのファイル保存
  - 使用 API や処理結果のサマリー出力
- 学び:
  - 長い処理は成功失敗だけでは足りず、「経過」が残っていることが重要

### 要約標準化

- 代表例:
  - `生成日時` `ステータス` `概要` `指標` を共通見出しにする
  - 実行ごとに `latest-summary.md` を更新する
- 学び:
  - 進捗ログと最終サマリーを分けると、確認コストが大きく下がる
  - 共通見出しを先に決めると、別案件への展開がしやすい

## 保留から採用に上げる条件

- 再実行時に同じ効果が確認できる
- 手作業より事故率が下がる
- 案件固有ではなく、別案件にも流用できる
- README や改善ログに短く説明できる形まで整理できる
