# 改善ログ

この案件で実施した改善と、他案件にも横展開できる学びを残すメモです。

## 案件概要

- 案件名: `CCキャンプ-リスト作成`
- 主な対象: CrowdWorks 系の営業リスト作成、YouTube 起点の企業・連絡先探索、シート書き込み自動化
- 主な出力先: Google スプレッドシート

## 詳細メモ

### 1. URL品質の改善

- 背景: 日本語を含む YouTube URL や handle URL で、開けない・再現しにくいものがあった
- 対応:
  - URLチェックと軽い自動修正を標準化
  - YouTube URL は `channel/UC...` 形式に正規化
  - 専用スクリプトで対象列を固定し、誤列更新を防止
- 学び:
  - URLは保存時に正規化した方が後工程が安定する
  - 列番号はマジックナンバーにせず定数化した方が安全

### 2. 会社名・代表者名補完の改善

- 背景: 代表者名に「代表取締役」などの役職が混ざり、会社名も拾い過ぎることがあった
- 対応:
  - 役職除去ロジックを導入
  - 信頼性チェックで短すぎる値やノイズを除外
  - 既存行を再評価する補完スクリプトを用意
- 学び:
  - 「抽出」と「整形」を分けると改善しやすい
  - 補完系は既存行を再走査できる形にしておくと運用が楽

### 3. メアド取得の改善

- 背景: 通常の問い合わせページだけではメール欄が埋まらないことがあった
- 対応:
  - 特商法ページ、会社概要、問い合わせ導線を優先探索
  - メール取得元をシートで選べるように整理
- 学び:
  - 「どこから取ったか」も一緒に残すと再確認しやすい
  - 取得元の分類は後から集計や見直しに効く

### 4. APIクォータ管理

- 背景: YouTube API の上限到達で処理が止まることがあった
- 対応:
  - 複数キーのフォールバック
  - 高コストな `search.list` 依存を減らし、低コスト API を優先
  - 使用量サマリーを出力
- 学び:
  - 精度だけでなく、API単価も設計対象
  - 「なぜ止まったか」を可視化しないと次回改善につながらない

### 5. シート更新の安全性

- 背景: A列の数式や色付きセルの仕組みを壊すと、手戻りが大きい
- 対応:
  - `A列は触らない`
  - `東たくみ` 以外の行は変更しない
  - 更新対象列を明示したスクリプトに分離
- 学び:
  - シート案件は「何を更新するか」より「何を触らないか」を先に決めるべき
  - dry-run か限定範囲指定があると事故が減る

### 6. ログ要約の標準化

- 背景: ログファイルは残っていても、スクリプトごとにサマリーの書き方が違い、見返しづらかった
- 対応:
  - 共通のサマリー書き出しヘルパーを追加
  - `生成日時` `ステータス` `概要` `指標` を共通見出しとして統一
  - URLチェック系、テスト系、YouTube探索系のサマリーを同じ型に寄せた
- 学び:
  - 長いログとは別に「最初に見る1枚」を固定すると作業再開しやすい
  - 共通フォーマットは実装より先に見出しを決めた方が広げやすい

## 今後この案件で標準化したいこと

- 実行前チェックリストの固定化
- URL正規化を保存前に必ず通す
- 取得元ラベルの候補をコード側とシート側で同期する
- 補完スクリプトに dry-run を増やす
